# STEP27-C 监控与应急预案（Incident / Monitoring Plan）

## A. 文档头信息与适用范围
- 项目：Xinyi_AI
- 阶段：STEP27-C（监控与应急预案）
- 版本范围：`main`（执行时记录 `git rev-parse --short HEAD`）
- 日期：2026-02-27
- 适用环境：Windows 11 + WSL2 Ubuntu 22.04 + MySQL 8.0
- 目标：在首批试运行/小规模商用阶段，确保“可发现、可定位、可止损、可回滚、可复盘”。
- 边界：本预案不引入新监控平台，采用现有日志 + 手工巡检 + 预定义应急流程。

## B. 监控目标与关键风险点（按业务）

### B1. 核心流程可用性
- 监控目标：关键页面/API 可访问，核心流程无持续 500。
- 关键风险：服务启动成功但核心接口报错（历史真实案例：S5/S6 SQL 500）。
- 最小检查点：
  - 页面：`/dashboard`
  - API：`/api/system/roles`、`/api/assets/changes`、`/api/assets/ledger`

### B2. 财务数据正确性风险（折旧、台账、报表）
- 监控目标：折旧计提与台账口径一致。
- 关键风险：
  - 折旧计提成功但台账累计折旧不一致
  - 期间筛选导致无折旧分录资产漏出
- 最小检查点：
  - `POST /api/assets/depreciation`
  - `GET /api/assets/ledger?book_id=<id>&dep_year=<y>&dep_month=<m>`

### B3. 资产变动与审计可追溯
- 监控目标：资产变动记录可查、关键字段完整（含 `note` / `operator`）。
- 关键风险：SQL 字段歧义导致查询失败（STEP26 已出现并修复）。
- 最小检查点：
  - `POST /api/assets/<id>/change`
  - `GET /api/assets/changes?book_id=<id>`

### B4. 数据库可用性
- 监控目标：数据库连接、迁移、写入稳定。
- 关键风险：WSL 到 Windows MySQL 不通、迁移失败、权限异常。
- 最小检查点：
  - `mysql -h127.0.0.1 -P3306 -u<user> -p -e "SELECT 1;" xinyi_ai`
  - `alembic current` / `alembic heads`

### B5. 导入导出可用性
- 监控目标：银行/税务导入与报表导出可用。
- 关键风险：导出失败阻断对账或复核。
- 最小检查点：
  - `POST /api/bank_transactions/import`
  - `GET /api/exports/assets/ledger?book_id=<id>`

## C. 日常巡检清单（最小版）

建议频率：工作日每日 2 次（开盘前/收盘后）。

- [ ] 服务进程可启动，`/` 返回 `{"status":"ok"}`
- [ ] 数据库连接检查 `SELECT 1` 成功
- [ ] 关键页面/API 冒烟：
  - [ ] `GET /dashboard` -> 200
  - [ ] `GET /api/assets/changes?book_id=<id>` -> 200
  - [ ] `GET /api/assets/ledger?book_id=<id>&dep_year=<y>&dep_month=<m>` -> 200
- [ ] 错误日志快速扫描（最近 24h 是否有连续 500 堆栈）
- [ ] 当日关键业务抽查 1 组：
  - 资产变动记录可查
  - 折旧与台账口径一致

## D. 故障分级（P0/P1/P2）

### D1. 定义
- `P0`：核心流程不可用或存在错账风险，需立即止损/可能回滚。
- `P1`：主要功能降级，存在业务影响但可短时人工规避。
- `P2`：非核心功能异常或体验问题，不影响主账务链路。

### D2. 例子
- `P0`：`/api/assets/ledger` 或 `/api/assets/changes` 持续 500；折旧/台账口径明显错误。
- `P1`：导出失败但核心记账/查询可继续；部分导入格式兼容问题。
- `P2`：页面样式或非关键提示异常。

### D3. 响应时限建议
- `P0`：15 分钟内响应，60 分钟内给出止损动作。
- `P1`：30 分钟内响应，当日给出修复或规避方案。
- `P2`：1 个工作日内响应，纳入后续版本计划。

## E. 应急响应流程（执行版）

### E1. 发现故障
- 记录：时间、路径、状态码、影响范围、触发操作。

### E2. 初步分类
- 按 D 节判定 P0/P1/P2。
- 若疑似错账风险，直接按 P0 处理。

### E3. 先止损
- P0 立即暂停相关业务操作（如资产变动/折旧/出账）。
- 对外发布临时告警：受影响模块 + 暂停范围 + 下一更新时间。

### E4. 收集证据
- 保存失败请求（路径、参数、请求头脱敏、响应体）。
- 保存日志/堆栈与 SQL 报错。
- 保存当前版本 commit 与迁移版本。

### E5. 定位修复
- 依 STEP27-B 排障路径定位到 service 层或环境配置。
- 修复范围控制在最小变更，避免引入额外风险。

### E6. 回归验证
- 至少覆盖：故障接口 + 1 条关联链路（例如 S4->S6）。
- 验证通过前不恢复暂停业务。

### E7. 恢复服务
- 恢复后持续观察至少 30 分钟。
- 更新事件状态与影响说明。

### E8. 复盘
- 24 小时内完成简版复盘（见 G 节模板）。

## F. 回滚与恢复协同

### F1. 回滚触发条件（引用 STEP27-A）
- 核心流程持续 500。
- 发现错账风险。
- 迁移失败或迁移后关键表异常。

### F2. 回滚前必须完成
- 证据留存（日志、请求、截图、版本、迁移号）。
- 最新备份可用性确认。
- 回滚决策人确认（技术负责人 + 业务负责人）。

### F3. 回滚后验证点
- 页面：`/dashboard` 200。
- API：`/api/assets/changes`、`/api/assets/ledger` 200。
- 账务口径：抽查 1 组折旧/台账一致性。
